# Exercise 00 — Roles and their activities `as is` (Task 2, prefix: del)

Источник ролей (каталог стейкхолдеров): `src/del/ex00_del_stakeholders.md` (st002001..st002014).

`As is` в задаче 2: единая онлайн-система распределения заказов и мобильное приложение ещё не внедрены; заказы поступают «любыми доступными способами», оператор приводит их к единому виду (вручную), распределение и контроль — через таблицы/звонки/мессенджеры.

Таблица ролей, их действий и проблем в текущем состоянии (`as is`):

| Идентификатор | Стейкхолдер | Действие | `as is`: удалённо (телефон/мессенджеры/почта/файлы) | `as is`: очно | Проблемы `as is` |
|---:|---|---|:---:|:---:|---|
| st002001 | Владелец(и) стартапа / Учредители | Принимать решения (модель работы, тарифы, KPI, приоритеты) | + | +/− | Решения принимаются без актуальных метрик; данные о выполнении/срывах собираются вручную и неполные |
| st002001 | Владелец(и) стартапа / Учредители | Контролировать операционную деятельность (скорость, качество, расходы) | + | - | Нет прозрачности по статусам заказов и загрузке курьеров; трудно масштабировать и улучшать процесс |
| st002002 | Оператор ввода заказов | Получать заказы от поставщиков в разном формате | + | - | Разнородные каналы/форматы → ошибки, потери данных, высокая нагрузка в пики |
| st002002 | Оператор ввода заказов | Нормализовать данные и вручную вводить в единый формат (таблица/учёт) | + | - | Ручной ввод медленный; сложно валидировать адреса/сроки; человеческий фактор |
| st002002 | Оператор ввода заказов | Уточнять недостающую информацию у поставщика/клиента | + | - | Уточнения занимают время; задерживают выдачу заказа курьеру; часть заказов «зависает» |
| st002003 | Курьер | Получать информацию о свободных заказах/назначениях | + | - | Нет единого списка актуальных заказов; информация устаревает; возможны конфликты «кто взял» |
| st002003 | Курьер | Бронировать заказ (сообщить, что берёт) | + | - | Бронирование через чат/звонок не атомарно; риск двойного назначения |
| st002003 | Курьер | Забрать заказ в точке комплектации/выдачи | - | + | Нет стандартных статусов/подтверждений; курьер может приехать раньше готовности; простои |
| st002003 | Курьер | Доставить заказ клиенту и подтвердить вручение | +/− | + | Подтверждение не унифицировано (нет POD-кода/фото/подписи); спорные ситуации решаются вручную |
| st002003 | Курьер | Передавать статусы (в пути/забрал/доставил/проблема) | + | - | Диспетчер/клиент видят статусы с задержкой; сложнее реагировать на сбои |
| st002004 | Диспетчер | Контролировать выполнение и статусы заказов | + | - | Статусы собираются из сообщений; нет общей картины; трудно отслеживать SLA/опоздания |
| st002004 | Диспетчер | Переназначать/перераспределять заказы | + | - | Много ручной коммуникации; ошибки при переназначении; сложно учитывать загрузку и приоритет |
| st002005 | Магазин / Предприятие питания (поставщик) | Передавать информацию о заказах и сроках готовности | + | - | Нет единого формата передачи; ошибки в адресах/товарах; сложно подтвердить, что заказ принят |
| st002005 | Магазин / Предприятие питания (поставщик) | Выдавать заказ курьеру в точке выдачи | - | + | Выдача может затягиваться из‑за очередей/неготовности; нет прозрачных статусов готовности |
| st002006 | Клиент (получатель заказа) | Оформлять заказ у поставщика, указывать адрес/время/контакт | + | - | Клиент не понимает, кто и когда доставит; сложно изменить адрес/время без множества звонков |
| st002006 | Клиент (получатель заказа) | Получать доставку и подтверждать получение | +/− | + | Нет трекинга и ETA; споры о «доставлено/не доставлено» решаются долго |
| st002011 | Служба поддержки клиентов | Принимать обращения по заказам (опоздание, отмена, замена) | + | - | Нет карточки заказа/истории статусов; приходится уточнять вручную у диспетчера/курьера |
| st002007 | Бухгалтерия (внешняя система) | Получать сведения о заказах для расчётов с поставщиками | + | - | Данные передаются файлами/вручную; несогласованные форматы; ошибки и задержки |
| st002007 | Бухгалтерия (внешняя система) | Получать сведения о доставках для расчёта оплаты курьеров | + | - | Нет автоматической фиксации фактов доставки; высокие риски расхождений и споров по выплатам |
| st002008 | Администратор системы | Регистрировать курьеров и назначать права доступа | +/− | +/− | До системы процесс ручной и несогласованный; сложно контролировать доступ и актуальность данных |
| st002010 | Провайдер платежей (payment gateway) | Обрабатывать оплату/возвраты и уведомлять о статусах | - | - | Оплата/возвраты проходят вне контура стартапа (у поставщика/наличными); трудно сверять оплату с фактом доставки и разбирать спорные случаи |
| st002013 | Партнёры по логистике / fulfilment центры | Комплектовать и передавать заказы курьерам (если задействованы) | + | + | Нет стандарта обмена статусами готовности; курьеры приезжают «впустую», растут задержки и простои |
| st002014 | Тестовые пользователи / ранние клиенты | Давать обратную связь по сервису/качеству доставки | + | +/− | Обратная связь несистемна (чаты/звонки); сложно агрегировать, приоритизировать и отслеживать улучшения |
| st002012 | Регуляторы / муниципальные службы | Проверять соблюдение норм (перевозка, санитария, ПДн) | - | +/− | Документы/журналы ведутся разрозненно; сложно быстро подготовить отчётность по запросу |
| st002009 | Команда разработки / DevOps | Собирать обратную связь и уточнять процессные требования | + | - | Слабая измеримость процесса: нет логов/метрик, требования формируются «со слов» и меняются |
